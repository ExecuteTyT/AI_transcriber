### P0-1 — Прод: регистрация и вход полностью сломаны (миграции не применены) ✅ ИСПРАВЛЕНО 2026-05-27

- **Где:** `POST /api/auth/register`, `POST /api/auth/login`, `POST /api/auth/request-password-reset` на `https://dicto.pro`.
- **Ожидание:** валидная регистрация → 201; вход несуществующего → 401.
- **Факт:** **500 Internal Server Error** (тело `text/plain`, 21 байт — дефолтный ответ Starlette, т.е. необработанное исключение приложения, не nginx). Воспроизводится 100%. Невалидный ввод корректно даёт 422 — т.е. 500 возникает именно при первом обращении к БД.
- **Корневая причина (подтверждена):** таблица `users` на проде **не содержит колонку `is_unlimited`**. Любой `SELECT User` / `INSERT User` ссылается на неё → ошибка БД → 500. Это значит, что на проде **не применены миграции** `k1f2a4b5c6d7_add_is_unlimited` (от 2026-05-21) и слитый head `m3b5c6d7e8f9_merge_mistral_and_main_chain`.
- **Доказательство (локальный repro):**
  - На корректной схеме: `login` несуществующего → **401**, `register` → **201**.
  - После `ALTER TABLE users DROP COLUMN is_unlimited`: `login` → **500**, `register` → **500** (идентично проду).
  - После восстановления колонки: снова **401 / 201**.
- **Влияние:** 0% пользователей могут зарегистрироваться/войти. Реклама = слив бюджета в мёртвую воронку. **Блокирует запуск.**
- **Исправление (на проде, по SSH):**
  ```bash
  ssh root@<prod-ip>
  cd /opt/aivoice
  docker compose -f docker-compose.prod.yml run --rm api alembic current   # покажет head ПОЗАДИ k1/m3
  docker compose -f docker-compose.prod.yml run --rm api alembic upgrade head
  # затем перепроверить:
  ./scripts/check-prod.sh https://dicto.pro
  ```
- **Статус:** ✅ исправлено в ходе прогона (`ALTER TABLE` + `alembic upgrade heads`). Register/login/refresh/consents проверены — 200.
- **Хвост (P1-3 ниже):** прод крутит устаревший образ; `alembic_version` теперь `a7b8 + j0`, колонка `is_unlimited` без записи `k1` → нужна нормальная пересборка + reconcile.

### P1-2 — Backend-тесты не запускаются (125 errors), замаскировано CI 🟠

- **Где:** `backend/tests/` + [.github/workflows/ci.yml](../.github/workflows/ci.yml) (`pytest ... || true`).
- **Факт:** `pytest` → **125 errors**, все `sqlalchemy.exc.UnsupportedCompilationError: can't render element of type INET`. Колонки типа Postgres `INET` (напр. `refresh_tokens.ip_address`) не компилируются на SQLite → `Base.metadata.create_all` падает в фикстуре `setup_db` → **ни один тест не исполняется**.
- **Почему важно:** фактически **нулевая авто-защита от регрессий** на backend. Это и есть причина, по которой дрейф миграций (P0) никто не поймал. CI «зелёный» из-за `|| true`.
- **Рекомендация:**
  1. Сделать INET-колонки совместимыми с SQLite: `INET().with_variant(sa.String(), "sqlite")` или `TypeDecorator`.
  2. Убрать `|| true` из CI — красные тесты должны блокировать merge.
  3. Добавить тест happy-path регистрации/логина именно против Postgres (как в CI-сервисе), чтобы дрейф схемы ловился.

### P2-1 — Загрузка принимает не-аудио по заявленному Content-Type (нет content-sniffing) 🟡

- **Где:** `POST /api/transcriptions/upload`.
- **Факт:** файл `fake.mp3` с текстом внутри, но с заголовком `Content-Type: audio/mpeg` → **201 принят**. Файл `.txt` (`text/plain`) → 400 (отклонён корректно).

- **Рекомендация:** валидировать по сигнатуре файла (magic bytes), а не по клиентскому MIME; либо явно ловить ошибку в пайплайне и помечать транскрипцию `failed` с понятным сообщением.
